# Entities Layer (SeaORM)

**Location:** `src/entities/`

Entities are the data layer of the application. They are generated by `sea-orm-codegen` and map directly to database tables. Each file defines the three core SeaORM types for a table.

For field-level detail and relationship diagrams, see the [Database Schema](../architecture/database-schema.md).

---

## SeaORM Concepts

### The Three Core Types

Every entity file defines:

| Type          | Purpose                                                              |
|---------------|----------------------------------------------------------------------|
| `Model`       | Read-only struct. Represents a row fetched from the database.        |
| `ActiveModel` | Writable struct used for INSERT and UPDATE. Fields are wrapped in `ActiveValue<T>`. |
| `Entity`      | The table descriptor. Used to build queries (`Entity::find()`, `Entity::find_by_id()`, etc.). |

Example from `accounts.rs`:

```rust
// Model ‚Äî returned by queries
pub struct Model {
    pub id: Uuid,
    pub email: String,
    // ...
}

// ActiveModel ‚Äî used for inserts/updates
let new_account = accounts::ActiveModel {
    id: Set(Uuid::new_v4()),
    email: Set(body.email),
    // unset fields use ..Default::default()
    ..Default::default()
};
new_account.insert(&state.db).await?;
```

### `ActiveValue`

Each field in an `ActiveModel` is an `ActiveValue<T>`:

| Variant         | Meaning                                                 |
|-----------------|---------------------------------------------------------|
| `Set(value)`    | Include this field in the SQL INSERT/UPDATE             |
| `NotSet`        | Omit this field (use the DB default or leave unchanged) |

### Derive Macros

```rust
#[derive(Clone, Debug, PartialEq, Eq, DeriveEntityModel, Serialize, Deserialize)]
#[sea_orm(table_name = "accounts")]
pub struct Model { ... }
```

- `DeriveEntityModel` ‚Äî generates `Entity`, `ActiveModel`, `Column`, and `Relation` from the `Model` struct.
- `Serialize, Deserialize` ‚Äî allows the model to be returned directly as a JSON response body.

## Relations

Relations are declared via the `Relation` enum and `impl Related<OtherEntity> for Entity` blocks. SeaORM uses these to enable join queries.

Cascade behaviour is set at the database level (in migrations) and mirrored in the `Relation` enum attributes (`on_delete = "Cascade"`).

### Quick Reference

| Entity         | Belongs To                    | Has Many                                        |
|----------------|-------------------------------|-------------------------------------------------|
| `accounts`     | ‚Äî                             | projects, tasks, answers, attachments, moderators |

## Common Query Patterns

```rust
use sea_orm::{EntityTrait, ColumnTrait, QueryFilter};
use crate::entities::{accounts, accounts::Entity as Account};

// Fetch all rows
let all_users = Account::find().all(&db).await?;

// Fetch by primary key
let user = Account::find_by_id(id).one(&db).await?;

// Filter by column
let user = Account::find()
    .filter(accounts::Column::Email.eq("user@example.com"))
    .one(&db)
    .await?;

// Insert
let new_row = accounts::ActiveModel {
    id: Set(Uuid::new_v4()),
    email: Set("user@example.com".to_string()),
    ..Default::default()
};
let inserted = new_row.insert(&db).await?;
```

## Migrations

Migrations live in the `migration/` workspace crate, which uses `sea-orm-migration`. Run migrations with:

```bash
sea-orm-cli migrate up
```

Entity files in `src/entities/` are regenerated from the live schema using:

```bash
sea-orm-cli generate entity -o src/entities
```

Do **not** hand-edit generated entity files; make schema changes in migrations and regenerate.
